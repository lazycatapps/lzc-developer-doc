# lzc-developer-doc Writing Constraints

本文档用于约束本仓库技术文档的写作方式。目标是让初学者可以快速执行、快速验证、快速排错。

## 1. 核心目标

1. 文档必须以“可执行”为第一优先级，而不是概念完整性。
2. 读者应在最短路径内完成第一个成功结果（先成功，再解释原理）。
3. 所有步骤应可复现，不依赖作者的隐含上下文。

## 2. 受众与语气

1. 默认受众是第一次接触 lpk 的开发者。
2. 语言要直接、技术化、去营销化，避免口号和空泛描述。
3. 对关键术语给出一次清晰定义，后续保持一致，不随意换称呼。

## 3. 文章结构（MUST）

每篇教程类文档应尽量包含以下结构：

1. 目标：这一篇完成后，读者能得到什么结果。
2. 前置条件：版本、依赖、环境要求。
3. 步骤：按顺序执行的命令和操作。
4. 验证：如何判断当前步骤成功。
5. 常见错误：至少给出最常见失败场景和排查命令。
6. 下一步：应该阅读哪一篇继续推进。

## 4. 命令与示例规范

1. 命令必须完整、可直接复制执行，不省略必要参数。
2. 下载源码必须写完整命令（例如 `git clone`），不能只给 URL。
3. 命令块使用 `bash` 或 `shell`，配置块使用 `yml` 或 `json`。
4. 命令输出示例必须与当前实际行为一致，不得沿用过期输出。
5. 如命令存在平台差异，必须明确按平台拆分。

## 5. 入门链路约束（MUST）

1. 入门路径优先：先成功部署第一个 lpk，并可在手机/PC访问验证。
2. 第二阶段：修改源码并重新部署验证变更。
3. 第三阶段：再引入后端代码与前后端协作。
4. 第四阶段：最后讲路由、manifest 等关键概念。
5. `devshell` 不作为入门路径，不放在新手主流程中。

## 6. 内容边界与一致性

1. 不要引导读者使用已经不推荐的命令或旧流程。
2. 不要在未说明前置条件时引入高级功能。
3. 不要把“可选功能”写成“必需步骤”。
4. 修改文档时应同步检查相关上下游页面，避免链路断裂。

## 7. 质量门禁（提交前检查）

1. 检查命令是否真实可执行，至少走查一遍主路径。
2. 检查文中版本、命令、输出示例是否自洽。
3. 检查是否存在过期关键词和历史路径残留。
4. 检查内部链接是否可跳转，章节引用是否存在。

## 8. 多语言策略

1. 默认先维护中文文档（`docs/`）。
2. 英文文档（`docs/en/`）仅在明确要求时同步改动。
