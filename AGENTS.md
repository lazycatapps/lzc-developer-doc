# lzc-developer-doc Writing Constraints

本文档用于约束本仓库技术文档的写作方式。目标是让初学者可以快速执行、快速验证、快速排错。

## 1. 核心目标

1. 文档必须以“可执行”为第一优先级，而不是概念完整性。
2. 读者应在最短路径内完成第一个成功结果（先成功，再解释原理）。
3. 所有步骤应可复现，不依赖作者的隐含上下文。

## 2. 受众与语气

1. 默认受众是第一次接触 lpk 的开发者。
2. 语言要直接、技术化、去营销化，避免口号和空泛描述。
3. 对关键术语给出一次清晰定义，后续保持一致，不随意换称呼。

## 3. 文章结构（MUST）

每篇教程类文档应尽量包含以下结构：

1. 目标：这一篇完成后，读者能得到什么结果。
2. 前置条件：版本、依赖、环境要求。
3. 步骤：按顺序执行的命令和操作。
4. 验证：如何判断当前步骤成功。
5. 常见错误：至少给出最常见失败场景和排查命令。
6. 下一步：应该阅读哪一篇继续推进。

## 4. 命令与示例规范

1. 命令必须完整、可直接复制执行，不省略必要参数。
2. 下载源码必须写完整命令（例如 `git clone`），不能只给 URL。
3. 命令块使用 `bash` 或 `shell`，配置块使用 `yml` 或 `json`。
4. 命令输出示例必须与当前实际行为一致，不得沿用过期输出。
5. 如命令存在平台差异，必须明确按平台拆分。

## 5. 入门链路约束（MUST）

1. 入门路径优先：先成功部署第一个 lpk，并可在手机/PC访问验证。
2. 第二阶段：修改源码并重新部署验证变更。
3. 第三阶段：再引入后端代码与前后端协作。
4. 第四阶段：最后讲路由、manifest 等关键概念。
5. `devshell` 不作为入门路径，不放在新手主流程中。

## 6. 内容边界与一致性

1. 不要引导读者使用已经不推荐的命令或旧流程。
2. 不要在未说明前置条件时引入高级功能。
3. 不要把“可选功能”写成“必需步骤”。
4. 修改文档时应同步检查相关上下游页面，避免链路断裂。

## 7. 质量门禁（提交前检查）

1. 检查命令是否真实可执行，至少走查一遍主路径。
2. 检查文中版本、命令、输出示例是否自洽。
3. 检查是否存在过期关键词和历史路径残留。
4. 检查内部链接是否可跳转，章节引用是否存在。

## 8. 多语言策略

1. 默认先维护中文文档（`docs/`）。
2. 英文文档（`docs/en/`）仅在明确要求时同步改动。

## 9. spec 与 advanced 文档分工（MUST）

1. `docs/spec/*.md` 只承担“规范定义”职责：字段、类型、默认值、约束、匹配规则与最小示例；要求精简、准确、完整，不展开实现细节推导。
2. `docs/advanced-*.md` 承担“进阶说明”职责：设计动机、行为细节、运行时机制、复杂示例、最佳实践、常见误区与排查方式。
3. `spec` 中涉及复杂语义时，应给出一段简要规则总结，并链接到对应 `advanced-*` 文档，不在 `spec` 内重复长篇讲解。
4. 功能语义变更时，必须同步检查 `spec` 与对应 `advanced-*` 是否一致；如当前功能尚未发布，不需要保留兼容性描述，直接以当前实现为准。

## 10. 术语规范（MUST）

1. 对外文档统一使用“微服”指代设备，不使用“盒子”作为产品术语。
2. 涉及 CLI 命令时保留命令原文（例如 `lzc-cli box ...`），但解释文本仍使用“微服”。

## 11. getting-started 专项约束（MUST）

1. `docs/getting-started/` 必须假设读者是刚接触懒猫微服开发的用户，不预设其具备 lpk、容器、路由背景。
2. 主流程应只包含最小必需步骤，优先“先跑通、再解释”，尽量不把高级功能放进必经路径。
3. 如确需引入高级功能，必须明确标注为“可选/进阶”，并在同一小节给出对应延伸阅读链接。
4. 首次出现技术术语时必须给出一句话定义，避免依赖内部上下文或历史文档记忆。

## 12. changelog 约束（MUST）

1. 默认禁止修改任何已发布版本对应的 changelog 内容。
2. 只有在用户再三明确要求时，才允许改动已发布版本的 changelog。

## 13. 文档锚点规范（MUST）

1. 文档中的显式锚点（例如 `{#...}`）统一使用英文（建议 `kebab-case`）。
2. 避免使用中文锚点，防止 URL 转义后过长影响可读性与分享体验。
